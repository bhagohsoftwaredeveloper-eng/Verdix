# Design: Multi-Writer SI Numbering

**Date:** 2026-07-06
**Status:** Approved (design), pending implementation plan
**Context:** Sub-project 1 of 5 in the "full two-way (multi-master) sync" decomposition.
The others (deferred, later specs): (2) full bidirectional pull, (3) delta-based
stock reconciliation, (4) conflict-resolution policy, (5) cash-drawer/shift +
Z/X-reading isolation. This sub-project is the BIR-critical foundation and must
land first.

## Problem

Sales-invoice (SI) numbers are a single global sequence, generated by
`getNextSINumber()` (lib/mysql.ts): an atomic `UPDATE transaction_references SET
si_number = si_number+1 WHERE id=1`, formatted 6-digit zero-padded (`001234`).
This is correct only because a store's terminals share ONE local DB, so there is
one counter. `transaction_references` is in the cloud-sync `EXCLUDE_TABLES`, so
each database keeps its own counter.

Once the web deployment (writing directly to the cloud DB) and each offline
desktop (its own local DB) both create sales, each generates from its own counter
→ they produce the SAME numbers (`001234` on both) → duplicate SI numbers on the
shared cloud table. BIR requires SI numbers to be sequential with no gaps and no
duplicates; this is legally significant.

## Decisions (locked)

1. **Per-machine series** (the BIR-native model): each writer issues invoices in
   its own series, identified by a unique **prefix**. Distinct prefixes make the
   full number globally unique with no cross-writer coordination — works offline.
2. **Manual prefix config per deployment**: an admin assigns each deployment a
   unique prefix; the app enforces format and fails loud if unset.
3. **Legacy invoices are left untouched** (historical plain numbers preserved for
   BIR); the main store gains a prefix going forward only. No backfill.

## Components

### ① Series prefix config (per-DB, not synced)
- Add column `si_prefix VARCHAR(8) NULL` to `transaction_references` (the row
  `id=1`). The table is already in `EXCLUDE_TABLES`, so each DB keeps its own
  prefix; the web's cloud counter and each store's local counter never interfere.
- Resolved at runtime from env `SI_SERIES_PREFIX`: on startup, if the stored
  `si_prefix` is empty and `SI_SERIES_PREFIX` is set, persist it; the stored value
  is authoritative thereafter. Web = `WEB`; each store = its own (`MAIN`, `BR2`…).
- Validation: prefix must match `^[A-Z0-9]{1,8}$`. If no prefix can be resolved
  (empty stored value AND empty env), `getNextSINumber()` throws — fail loud, so a
  blank prefix can never silently collide with another writer's blank prefix.

### ② `getNextSINumber()` becomes series-aware
- Still atomic per-DB inside `withTransaction`: increment the numeric counter as
  today (`UPDATE transaction_references SET si_number = si_number+1 WHERE id=1`,
  read it back).
- Returns the FULL invoice number `${si_prefix}-${LPAD(counter,6,'0')}` — e.g.
  `WEB-000045`, `MAIN-001234`. The numeric counter stays plain in
  `transaction_references.si_number`; the composed value is what callers store.
- The full prefixed value is written to `sales_transactions.si_number`,
  `sales_invoices.si_number`, `pos_transactions.si_number` (unchanged call sites in
  app/api/pos/checkout/route.ts). Because prefixes differ, these are globally
  unique across writers.

### ③ Format / validation
- `validateSINumber()` accepts the new form `^[A-Z0-9]{1,8}-\d{6}$` AND the legacy
  form `^\d{1,6}$` (so historical rows still validate).
- `formatSINumber()` passes a value through unchanged if it already contains a
  prefix (`-`), else pads to 6 digits (legacy behavior).
- Receipts, reports, search already source from `si_number`, so they display the
  full prefixed number with no per-call-site change.

### ④ Uniqueness safety net
- Add a UNIQUE index on `sales_invoices.si_number` — the canonical BIR invoice
  record, where each issued number must be unique. A misconfigured duplicate
  prefix then surfaces as a hard unique-key error on insert/sync instead of
  silently duplicating.
- The migration MUST first check for existing duplicate `si_number` values (there
  should be none under the current single-writer sequence) and abort with a clear
  message if any exist, rather than failing mid-index-build.
- `sales_transactions` / `pos_transactions` are deliberately NOT given a unique
  `si_number` index here: those tables may legitimately hold multiple rows sharing
  an SI number (e.g. a sale plus its void/return referencing the original). The
  plan will confirm this before deciding; the guard lives on `sales_invoices`.

### ⑤ Legacy continuity
- Existing rows keep their plain numbers; no backfill. The main store's counter
  continues from its current value, now emitting `MAIN-<counter>` going forward.
  The unique index tolerates the mix (legacy plain values are still distinct from
  new prefixed values).

## Data Flow

```
Web checkout       → getNextSINumber() → WEB-000045  → cloud sales_invoices
Desktop (offline)  → getNextSINumber() → MAIN-001234 → local → pushed to cloud later
Sync: WEB-* vs MAIN-* are distinct → no collision on the shared cloud table
```

## Error Handling

- Unresolvable/invalid prefix → `getNextSINumber()` throws at generation time
  (fail loud); checkout surfaces the error rather than issuing an ambiguous number.
- Duplicate full SI number (misconfigured identical prefixes) → unique-key error
  on `sales_invoices` insert/sync, caught and logged; never a silent duplicate.
- Offline → generation is fully local (counter + local prefix); no central
  dependency.

## Testing

- **Unit:** `getNextSINumber` composes `PREFIX-NNNNNN`; `validateSINumber` accepts
  new and legacy forms and rejects malformed; prefix validation regex; two
  distinct prefixes can never produce equal full numbers for equal counters.
- **Integration:** two DBs with different prefixes both insert/sync into one cloud
  `sales_invoices` with no duplicate-key error; two DBs with the SAME prefix and
  same counter → the unique index rejects the second (proving the safety net).

## Out of Scope (later sub-projects)

Full bidirectional pull, delta-based stock, conflict-resolution policy,
cash-drawer/shift isolation, Z/X-reading numbering. This spec changes ONLY SI
number generation, format/validation, and the uniqueness guard.
